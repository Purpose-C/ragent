## 1. 三个版本的项目介绍

### 1.1 1 分钟版本（电梯演讲）

**适用场景**：简历筛选、电话初面、开场破冰。

> Ragent 是一个基于 Java 17 + Spring Boot 3.5 的企业级 Agentic RAG 平台。它把企业私有知识通过文档入库 Pipeline 解析、切块、Embedding 后存进 PostgreSQL pgvector；用户提问时，系统会改写问题、识别意图、做多路召回和 Rerank，再流式调用大模型，通过 SSE 返回答案。项目还接入了 MCP 工具协议，能调用天气、工单等实时业务工具。模型层有候选优先级、三态熔断和首包探测等容错机制。

**必须提到的三个点**：
1. **企业级 RAG**：不是简单调大模型，而是有完整的知识入库、检索、生成链路。
2. **Agentic 能力**：问题改写、意图识别、MCP 工具调用、多路召回。
3. **可靠性**：模型路由、熔断降级、Trace 可观测。

### 1.2 3 分钟版本（常规面试）

**适用场景**：技术一面、项目深挖开场。

> Ragent 是一个前后端分离的企业级 Agentic RAG 平台，后端用 Java 17 + Spring Boot 3.5，前端用 React 18 + Vite + Zustand。
>
> 项目分成四个 Maven 模块：`framework` 提供统一响应、异常、上下文、幂等；`infra-ai` 屏蔽模型供应商差异，提供 `LLMService`、`EmbeddingService`、`RerankService`；`bootstrap` 是业务入口，编排 RAG 问答和文档入库；`mcp-server` 是独立的工具服务。
>
> 文档入库有两条链路：知识库文档上传后通过 RocketMQ 异步分块；通用 Ingestion Pipeline 在当前请求线程同步执行 `IngestionEngine`，按 Pipeline 定义跑 Fetcher、Parser、Chunker、Indexer 等节点，最终写入 `t_knowledge_chunk` 和 `t_knowledge_vector`。
>
> 问答流程从 `RAGChatController.chat()` 进入，`RAGChatServiceImpl.streamChat()` 生成 `taskId` 并进入 `StreamChatPipeline.execute()`，依次加载记忆、改写问题、识别意图、检索/MCP、组装 Prompt、流式调用模型，最后通过 `StreamChatEventHandler` 发送 SSE 事件。
>
> 模型调用不是硬编码单供应商。`RoutingLLMService` 会根据 `ModelSelector` 选候选，`ModelHealthStore` 维护 CLOSED/OPEN/HALF_OPEN 三态熔断，`ModelRoutingExecutor` 负责失败降级。流式 Chat 还做了首包探测，首包失败可以安全切换下一个候选。

### 1.3 10 分钟版本（详细项目介绍）

**适用场景**：技术二面、项目主管面、需要深度交流时。

> （前 3 分钟与 3 分钟版本相同）
>
> 我想重点讲三个设计点：
>
> **第一，入库 Pipeline 的可配置性。** 系统不是把解析、切块、Embedding 逻辑写死在代码里，而是用 `PipelineDefinition` + `NodeConfig` 描述节点连线。`IngestionEngine` 会校验是否有环、是否能找到起点，然后链式执行。每个节点执行结果生成 `NodeLog`，最终写入 `t_ingestion_task_node`。这样新增一种文档处理节点，只需要实现 `IngestionNode` 接口并注册，不需要改引擎主逻辑。
>
> **第二，RAG 问答的多阶段编排。** `StreamChatPipeline.execute()` 是核心，它按顺序调用 `loadMemory()`、`rewriteQuery()`、`resolveIntents()`、`retrieve()`、`streamRagResponse()`。所有中间状态保存在 `StreamChatContext` 里，包括 `RewriteResult`、`SubQuestionIntent`、`RetrievalContext`、`PromptContext`。这种 Context 模式让异步流式场景下的状态传递更清晰，也便于 Trace 埋点。
>
> **第三，模型路由与容错。** 业务代码只依赖 `LLMService` 接口，实际注入的是 `RoutingLLMService`。同步调用走 `ModelRoutingExecutor.executeWithFallback()`，遍历候选直到成功；流式调用因为不能简单重试，所以用 `ProbeStreamBridge` 做首包探测，首包失败才切换。`ModelHealthStore` 用失败次数和打开时长实现类熔断器状态机，避免持续调用失败的模型。
>
> 这三点结合起来，Ragent 在工程上做到了“可配置、可观测、可容错”。

## 2. 专项讲法

### 2.1 文档入库专项讲法

**核心入口**：
- `KnowledgeDocumentController.upload()`：`POST /knowledge-base/{kbId}/docs/upload`，保存文件到 RustFS/S3，插入 `t_knowledge_document`。
- `IngestionTaskController.upload()`：`POST /ingestion/tasks/upload?pipelineId=...`，同步执行 Pipeline。
- `IngestionEngine.execute()`：链式执行节点，生成 `NodeLog`。

**核心类与方法**：
- `IngestionEngine.execute(PipelineDefinition, IngestionContext)`
- `IngestionEngine.executeNode(IngestionContext, NodeConfig)`
- `IngestionTaskServiceImpl.saveNodeLogs()` → 写入 `t_ingestion_task_node`
- `IngestionTaskServiceImpl.updateTaskFromContext()` → 更新 `t_ingestion_task`

**核心表**：
- `t_knowledge_document`：文档元数据与状态。
- `t_knowledge_chunk`：Chunk 正文。
- `t_knowledge_vector`：Chunk 向量，字段 `embedding vector(1536)`。
- `t_ingestion_task` / `t_ingestion_task_node`：通用 Ingestion 任务与节点日志。

**1 分钟讲法**：
> 文档入库把 PDF/Word/TXT 通过 Fetcher 取文件、Parser 解析、Chunker 切块、Indexer 写向量，最终存到 PostgreSQL pgvector。知识库链路是异步 RocketMQ，通用 Ingestion 链路是同步 Pipeline。

**3 分钟讲法**：
> 入库由 `IngestionEngine` 编排，节点配置来自 `PipelineDefinition`。`NodeLog` 记录每个节点耗时和结果，落库到 `t_ingestion_task_node`。Chunker 调用 `ChunkEmbeddingService.embed()` 生成向量，Indexer 把向量写入 `t_knowledge_vector`。知识库链路还会同步写 `t_knowledge_chunk`。

### 2.2 RAG 问答专项讲法

**核心入口**：
- `RAGChatController.chat()`：`GET /rag/v3/chat`，返回 `SseEmitter`。
- `RAGChatServiceImpl.streamChat()`：生成 `taskId`、排队、创建 Trace。
- `StreamChatPipeline.execute()`：主编排。

**核心类与方法**：
- `StreamChatPipeline.execute(StreamChatContext, StreamCallback)`
- `DefaultConversationMemoryService.load()` / `append()`
- `MultiQuestionRewriteService.rewriteWithSplit()`
- `IntentResolver.resolve()`
- `RetrievalEngine.retrieve()`
- `RAGPromptService.buildStructuredMessages()`
- `RoutingLLMService.streamChat()`
- `StreamChatEventHandler.onContent()` / `onComplete()`

**核心表**：
- `t_conversation` / `t_message`：会话与消息。
- `t_conversation_summary`：长会话摘要。
- `t_rag_trace_run` / `t_rag_trace_node`：问答 Trace。

**关键配置**：
- `rag.memory.history-keep-turns=4`
- `rag.memory.summary-start-turns=5`
- `rag.default.sse-timeout-ms=300000`

**1 分钟讲法**：
> 用户提问后，系统加载历史、改写问题、识别意图，然后检索知识库或调用 MCP 工具，把结果塞进 Prompt，流式调用大模型，通过 SSE 逐字返回。

**3 分钟讲法**：
> 主流程在 `StreamChatPipeline.execute()` 里。先 `loadMemory()` 读近期历史，超过轮次后读摘要；然后 `rewriteQuery()` 把口语问题改写成检索友好形式；`resolveIntents()` 判断走知识库还是 MCP；`retrieve()` 做向量召回和 Rerank；最后 `RAGPromptService` 组装 Prompt，`RoutingLLMService.streamChat()` 流式生成，`StreamChatEventHandler` 把增量推给前端并持久化到 `t_message`。

### 2.3 检索与 Rerank 专项讲法

**核心类与方法**：
- `RetrievalEngine.retrieve()`
- `MultiChannelRetrievalEngine.retrieveKnowledgeChannels()`
- `SearchChannelProperties`
- `RoutingRerankService.rerank()`
- `DeduplicationPostProcessor`

**核心配置**：
- `rag.search.channels.vector-global.confidence-threshold=0.6`
- `rag.search.channels.vector-global.top-k-multiplier=3`
- `rag.search.channels.intent-directed.enabled=true`
- `rag.rerank.enabled=true`

**核心表**：
- `t_knowledge_vector`
- `t_intent_tree` / `t_intent_node`：意图定向范围。

**1 分钟讲法**：
> 检索不只是做一次向量搜索。系统先按意图定向召回，再做全局向量兜底，然后去重、Rerank，最后取 Top-K 送进 Prompt。

**3 分钟讲法**：
> `RetrievalEngine` 调用 `MultiChannelRetrievalEngine` 并行跑多个通道。`vector-global` 跨所有知识库兜底，`intent-directed` 在高置信度 KB 意图下只在目标知识库检索。召回结果经过 `DeduplicationPostProcessor` 去重，再经过 `RoutingRerankService` 重排。Rerank 默认走百炼 `qwen3-rerank`，失败会 fallback 到 `rerank-noop` 直接返回前 Top-K。

### 2.4 模型路由与容错专项讲法

**核心类与方法**：
- `LLMService.chat()` / `streamChat()`
- `RoutingLLMService.chat()` / `streamChat()`
- `ModelSelector.selectChatCandidates()`
- `ModelRoutingExecutor.executeWithFallback()`
- `ModelHealthStore.markFailure()` / `markSuccess()` / `allowCall()`

**核心配置**：
- `ai.chat.default-model=qwen3-max`
- `ai.chat.deep-thinking-model=qwen3-max`
- `ai.selection.failure-threshold=2`
- `ai.selection.open-duration-ms=30000`

**1 分钟讲法**：
> 业务代码只依赖 `LLMService` 接口。`RoutingLLMService` 选候选、做熔断、失败时降级。流式 Chat 还做首包探测，首包失败再切下一个模型。

**3 分钟讲法**：
> `ModelSelector` 根据 `thinking` 标志和候选的 `supports-thinking` 过滤，按 `priority` 排序。`ModelHealthStore` 维护 CLOSED/OPEN/HALF_OPEN 三态，失败次数超过 `failure-threshold` 就 OPEN，经过 `open-duration-ms` 后进入 HALF_OPEN。同步调用由 `ModelRoutingExecutor` 遍历候选；流式调用自己实现降级循环，用 `ProbeStreamBridge` 等首包，首包成功才返回 `StreamCancellationHandle`。

### 2.5 MCP 工具调用专项讲法

**核心类与方法**：
- `McpServerApplication.main()`（端口 9099）
- `WeatherMcpExecutor`、`TicketMcpExecutor`、`SalesMcpExecutor`
- `McpClientAutoConfiguration.registerRemoteTools()`
- `DefaultMcpToolRegistry`
- `IntentResolver.resolve()`
- `LLMMcpParameterExtractor`
- `McpClientToolExecutor.execute()`

**核心配置**：
- `rag.mcp.servers[0].url=http://localhost:9099`

**核心表**：
- `t_intent_tree` / `t_intent_node`：意图与 `mcp_tool_id` 关联。

**1 分钟讲法**：
> MCP 让 Ragent 能调用外部实时工具，比如天气、工单。知识问题走检索，实时业务问题走 MCP，结果都进 Prompt。

**3 分钟讲法**：
> `mcp-server` 是独立 Spring Boot 服务，暴露 `weather_query`、`ticket_query`、`sales_query` 三个工具。`bootstrap` 启动时通过 MCP SDK 调 `tools/list` 发现工具，注册到 `DefaultMcpToolRegistry`。问答时 `IntentResolver` 判断意图类型，MCP 意图走 `LLMMcpParameterExtractor` 提取参数，再经 `McpClientToolExecutor.callTool()` 调用远端工具，结果写入 `RetrievalContext.mcpContext`，最终进入 Prompt。

### 2.6 前端 SSE 专项讲法

**核心类与方法**：
- `frontend/src/hooks/useStreamResponse.ts`：`createStreamResponse()`、`readSseStream()`
- `frontend/src/stores/chatStore.ts`：`sendMessage()`
- `RAGChatController.chat()`
- `StreamChatEventHandler.onMeta()` / `onMessage()` / `onFinish()` / `onComplete()`

**核心配置**：
- `VITE_API_BASE_URL`
- `frontend/vite.config.ts` proxy：`/api` → `http://localhost:9090`

**核心 SSE 事件**：
- `meta`：携带 `conversationId`、`taskId`。
- `message`：类型 `think` / `response`。
- `finish`：携带数据库 `messageId`。
- `done`：流结束。
- `error`：发生错误。

**1 分钟讲法**：
> 前端用 fetch 读取 SSE 流，解析 `event:` 和 `data:` 行，把增量内容拼到 `chatStore.messages` 里，实现逐字显示。

**3 分钟讲法**：
> 因为原生 `EventSource` 不方便带 `Authorization` 头，项目用 fetch + `response.body.getReader()` 手动解析。`chatStore.sendMessage()` 先乐观插入用户消息和空 assistant 消息，拿到 `meta` 后更新 `taskId`，后续 `message` 事件按 `streamingMessageId` 拼接，`finish` 事件把临时 ID 替换成数据库真实 ID。

### 2.7 数据库设计专项讲法

**核心表分组**：
- 用户与认证：`t_user`
- 会话与消息：`t_conversation`、`t_message`、`t_conversation_summary`
- 知识库与文档：`t_knowledge_base`、`t_knowledge_document`
- Chunk 与向量：`t_knowledge_chunk`、`t_knowledge_vector`

**关键设计点**：

- `t_knowledge_vector.metadata` JSONB 存 `knowledgeBaseId`、`docId`、`chunkIndex`。
- `t_ingestion_task.logs_json` 和 `t_ingestion_task_node` 双写节点日志。

**1 分钟讲法**：
> 数据库用 PostgreSQL + pgvector，结构化数据存在关系表里，语义向量存在 pgvector 的 `t_knowledge_vector`，所有核心流程都有 Trace 或节点日志记录。

**核心类与方法**：
- `StreamChatTraceRunner.run()`
- `RagTraceAspect.aroundNode()`
- `RagStreamTraceSupportImpl.beginStreamNode()`
- `RagTraceRecordServiceImpl.startRun()` / `finishNode()`
- `RagTraceQueryServiceImpl.pageRuns()` / `detail()`
- `IngestionEngine.executeNode()` 中的 `NodeLog` 生成

**核心表**：
- `t_rag_trace_run` / `t_rag_trace_node`
- `t_ingestion_task` / `t_ingestion_task_node`

**核心配置**：

**1 分钟讲法**：

## 3. 面试题与参考答案

### 3.1 基础概念（6 题）

**Q1：什么是 RAG？和普通的大模型调用有什么区别？**

> RAG 是检索增强生成。普通调用直接把用户问题给模型，模型只依赖训练知识；RAG 先从知识库检索相关文档片段，再把片段放进 Prompt，让模型基于检索到的内容回答。Ragent 中检索阶段在 `RetrievalEngine.retrieve()`，Prompt 组装在 `RAGPromptService.buildStructuredMessages()`。

**Q2：什么是 Embedding？项目中哪里用到？**

> Embedding 把文本变成固定维度的浮点向量。Ragent 中两次用到：入库时 `ChunkerNode` 通过 `ChunkEmbeddingService.embed()` 给 Chunk 生成向量，写入 `t_knowledge_vector`；问答时 `PgRetrieverService` / `MilvusRetrieverService` 把改写后的问题也转成向量，做相似度检索。配置维度是 `rag.default.dimension=1536`。

**Q3：什么是 Agentic RAG？Ragent 有哪些 Agentic 能力？**

> Agentic RAG 是在普通 RAG 基础上增加主动决策能力，如问题改写、意图识别、工具调用、分支决策。Ragent 中 `MultiQuestionRewriteService` 做问题改写，`IntentResolver` 做意图识别，`McpClientToolExecutor` 做 MCP 工具调用，`StreamChatPipeline` 做整体编排。

**Q4：SSE 和 WebSocket 有什么区别？为什么 Ragent 用 SSE？**

> SSE 是服务端单向推送，基于 HTTP，适合模型逐字返回；WebSocket 是双向全双工。Ragent 只需要服务端向浏览器推流，SSE 更简单，且天然兼容 HTTP 代理和认证。代码中 `useStreamResponse.ts` 用 fetch 读取 SSE，而不是原生 `EventSource`，因为原生 API 不方便设置 `Authorization` 头。

**Q5：什么是 Rerank？为什么需要它？**

> Rerank 是对召回候选重新排序，用更强的相关性模型把最相关的片段排到前面。召回阶段重覆盖率，可能拿回很多候选；Rerank 把 Top-K 限制内的候选质量提高。Ragent 中 `RoutingRerankService.rerank()` 实现，默认候选是百炼 `qwen3-rerank`，失败会 fallback 到 `rerank-noop`。

**Q6：什么是 MCP？Ragent 中 MCP 和知识库检索是什么关系？**

> MCP 是模型上下文协议，让 AI 应用统一发现和调用外部工具。Ragent 中 MCP 不是替代知识库检索，而是补充：知识问题走 `RetrievalEngine` 检索文档，实时业务问题走 `McpClientToolExecutor` 调用工具，两者结果都写入 `RetrievalContext`，最终都进入 Prompt。

### 3.2 项目架构（6 题）

**Q7：Ragent 后端为什么分成 bootstrap、framework、infra-ai 三个模块？**

> `framework` 放通用 Web 能力：统一响应 `Result`、全局异常处理 `GlobalExceptionHandler`、用户上下文 `UserContext`、幂等 AOP `@IdempotentSubmit`、MQ 适配器。`infra-ai` 屏蔽模型供应商差异，提供 `LLMService`、`EmbeddingService`、`RerankService` 三个能力接口。`bootstrap` 是业务入口，编排 RAG、入库、用户、管理后台。这样换模型只改 `infra-ai`，通用能力不重复实现。

**Q8：mcp-server 为什么是独立模块，而不是 bootstrap 的子模块？**

> MCP Server 需要独立进程和端口（默认 9099）。如果放在 bootstrap 里，工具代码和业务代码强耦合，无法独立部署，也不符合 MCP 协议中“远端工具服务”的语义。`bootstrap` 通过 HTTP Streamable 方式连接 `mcp-server`。

**Q9：前端为什么不是 Maven 模块？**

> 前端是 React 项目，构建工具是 Vite，依赖 npm。它通过 HTTP/SSE 与后端通信，不是 Java 编译依赖，因此不需要也不应该作为 Maven 子模块。`frontend/package.json` 独立管理依赖。

**Q10：PostgreSQL、pgvector、RustFS/S3、Milvus 分别存什么？**

> PostgreSQL 存结构化业务数据：用户、会话、消息、文档元数据、Chunk 正文、Pipeline、Trace。pgvector 是 PG 扩展，存 `t_knowledge_vector.embedding`。RustFS/S3 存上传的源文件二进制。Milvus 是可选的独立向量存储，默认不启用，由 `rag.vector.type` 选择 `pg` 或 `milvus`。

**Q11：Ragent 的认证流程是怎样的？**

> 前端登录调用 `AuthController.login()`，Sa-Token 生成 Token 存 Redis。后续请求带 `Authorization` 头，`SaInterceptor` 校验登录，`UserContextInterceptor` 从数据库查用户并写入 `UserContext`。Controller 和 Service 通过 `UserContext.getUserId()` 获取当前用户。请求结束后 `afterCompletion` 清理 `UserContext`。

**Q12：统一返回对象 `Result<T>` 的约定是什么？**

> `Result` 包含 `code`、`message`、`data`、`requestId`。成功时 `code="0"`；失败时 `code` 是错误码字符串，如 `A000001`，HTTP 状态码仍为 200。前端 `api.ts` 响应拦截器判断 `payload.code !== "0"` 时 reject。SSE 接口不走 `Result` 包装。

### 3.3 源码流程（8 题）

**Q13：一次问答从用户输入到 SSE 返回，经过了哪些核心类？**

> 前端：`ChatInput.handleSubmit()` → `chatStore.sendMessage()` → `useStreamResponse.createStreamResponse()`。后端：`RAGChatController.chat()` → `RAGChatServiceImpl.streamChat()` → `ChatQueueLimiter` → `StreamChatTraceRunner` → `StreamChatPipeline.execute()` → 记忆/改写/意图/检索/Prompt/模型 → `StreamChatEventHandler` → SSE。

**Q14：`StreamChatPipeline.execute()` 里主要做了哪些事？**

> 按顺序调用：`loadMemory()` 加载历史；`rewriteQuery()` 改写问题；`resolveIntents()` 识别意图；`handleGuidance()` 处理澄清；`retrieve()` 检索或 MCP；`handleEmptyRetrieval()` 处理空检索；`streamRagResponse()` 调用模型并流式输出。所有中间状态保存在 `StreamChatContext`。

**Q15：`IngestionEngine` 如何保证 Pipeline 没有环？**

> `validatePipeline()` 遍历每个节点，用 `Set<String> path` 记录当前路径，如果发现 `path.contains(current)` 就说明有环，抛 `ClientException`。验证通过后 `findStartNode()` 找没有被任何节点引用的节点作为起点。

**Q16：`NodeLog` 是什么时候生成、什么时候落库的？**

> `IngestionEngine.executeNode()` 中三处生成：条件跳过时、节点成功执行后、节点抛异常时。节点执行完成后，`IngestionTaskServiceImpl.executeInternal()` 调用 `saveNodeLogs()` 把内存中的 `List<NodeLog>` 写入 `t_ingestion_task_node`，再调用 `updateTaskFromContext()` 把 `logs_json` 摘要写入 `t_ingestion_task`。

**Q17：模型路由如何选择候选？**

> `ModelSelector.selectChatCandidates(request.getThinking())` 根据是否深度思考过滤候选，再按 `priority` 排序。如果请求指定了 `modelId`，只构建单候选。同步调用由 `ModelRoutingExecutor.executeWithFallback()` 遍历候选，每个候选调用 `ModelHealthStore.allowCall()` 检查熔断状态，成功则返回，失败则继续下一个。

**Q18：流式 Chat 为什么不能用简单重试？**

> 因为模型已经开始向用户输出内容，如果直接重试，用户可能看到重复开头或半截回答。Ragent 用首包探测：在第一个 token 推给前端之前，用 `ProbeStreamBridge` 等待首包，首包失败才安全切换到下一个候选。首包成功后就返回 `StreamCancellationHandle`，不再切换。

**Q19：MCP 工具结果如何进入最终 Prompt？**

> `IntentResolver` 识别出 MCP 意图后，`LLMMcpParameterExtractor` 从用户问题提取参数，`McpClientToolExecutor.execute()` 调用远端工具得到 `CallToolResult`。结果写入 `RetrievalContext.mcpContext`，`RAGPromptService.buildStructuredMessages()` 根据 KB/MCP/混合场景选择不同模板（`answer-chat-kb.st`、`answer-chat-mcp.st` 等），把工具结果和检索结果一起塞进 Prompt。

**Q20：前端如何解析 SSE 事件？**

> `useStreamResponse.readSseStream()` 用 `fetch` 获取 `response.body.getReader()`，逐块读取字节并用 `TextDecoder` 解码。用缓冲区保存未完整行，遇到 `\n\n` 表示一个 SSE 事件结束，解析 `event:` 和 `data:` 行，调用 `onMeta` / `onMessage` / `onFinish` / `onComplete` / `onError` 回调。

### 3.4 设计取舍（5 题）

**Q21：为什么要做问题改写？改写失败怎么办？**

> 多轮对话中用户问题常有省略和指代，直接检索效果差。`MultiQuestionRewriteService.rewriteWithSplit()` 把问题改写成独立、清晰的检索句，并可能拆成多个子问题。改写失败时退回术语归一化后的问题，作为唯一子问题继续流程，不会中断问答。

**Q22：为什么要做多路召回？**

> 单一向量检索不擅长精确词匹配，且不同知识库范围下召回质量不同。`MultiChannelRetrievalEngine` 并行跑 `vector-global` 和 `intent-directed`，前者跨库兜底，后者按意图精准召回，融合后提高覆盖率。代价是去重和 Rerank 计算量增加。

**Q23：为什么 RAG Trace 不记录 `CANCELLED` 状态？**

> 当前 `StreamChatTraceRunner` 在用户取消时依赖回调的 `onFinish(success, error)` 收尾，最终状态取决于取消前是否已触发 `onError`。`RagStreamTraceSupportImpl` 内部有 `CANCELLED` 状态，但 RAG Trace Run 记录没有单独区分。这是当前实现的现状，面试中不要过度夸大。

**Q24：为什么 `t_ingestion_task.logs_json` 要去掉 `output` 字段？**

> 节点 `output` 可能包含大段文本、向量或结构化数据，直接序列化会让 `logs_json` 体积膨胀，且可能泄露原文。`IngestionTaskServiceImpl.buildLogSummary()` 把 `output` 置空后再写 `logs_json`，完整输出保留在 `t_ingestion_task_node.output_json` 中。

**Q25：为什么知识库文档链路和通用 Ingestion 链路要分开？**

> 知识库文档面向真实业务对象，需要持久化 `t_knowledge_document`、`t_knowledge_chunk`，并支持异步分块和重试。通用 Ingestion 面向 Pipeline 调试和通用数据摄取，同步执行、快速反馈、节点日志更详细。两者复用 `IngestionEngine` 节点逻辑，但写入职责不同。

### 3.5 排错场景（6 题）

**Q26：用户反馈“回答慢”，你怎么排查？**

> 先让前端提供 `taskId`，查 `t_rag_trace_run` 看总耗时和 `USER_TTFT` 节点。再看 `t_rag_trace_node` 各阶段耗时，定位是改写慢、检索慢、Rerank 慢还是模型慢。如果 `USER_TTFT` 长，重点看检索和模型首包；如果总耗时长但 TTFT 正常，看模型输出是否太长。同时检查后端日志是否有模型候选切换或超时。

**Q27：文档上传成功但问答时检索为空，可能是什么原因？**

> 先确认文档 `status` 是否为 `success`，`t_knowledge_chunk` 和 `t_knowledge_vector` 是否有记录。再确认问题对应的知识库是否正确。然后看 `confidence-threshold=0.6` 是否过高导致没有 chunk 过线。最后看问题改写后语义是否偏移，可尝试用原始问题直接做向量检索验证。

**Q28：MCP 工具没被调用，怎么排查？**

> 先确认 MCP Server 是否启动，用 curl 调 `tools/list`。再确认 `rag.mcp.servers` 配置正确。然后看 `IntentResolver` 是否识别出 MCP 意图，查 `t_intent_node` 中是否配置了 `mcp_tool_id`。最后看 `LLMMcpParameterExtractor` 是否成功提取参数，`McpClientToolExecutor.execute()` 是否被调用。

**Q29：模型一直超时，怎么排查？**

> 先看后端日志是否有 `first packet timeout` 或 `RemoteException`。再查 `ai.chat.candidates` 配置，确认 API Key 是否设置。然后在 `RoutingLLMService.streamChat()` 打断点，看候选列表和熔断状态。最后检查网络是否可达模型服务，以及 `rag.default.sse-timeout-ms` 是否设置过短。

**Q30：SSE 中断，怎么判断是前端还是后端问题？**

> 打开浏览器 DevTools → Network → `/rag/v3/chat` → EventStream，看最后收到的事件。如果最后事件是 `error`，看后端的异常日志；如果是直接断开，可能是 SSE 超时或网络代理问题；如果前端没有发请求，看 `chatStore` 状态或代理配置。同时查 `t_message` 是否有助手消息记录，判断后端是否已处理。

**Q31：Ingestion Pipeline 中 Parser 节点失败，怎么定位？**

> 查 `t_ingestion_task_node` 中 `node_type='parser'` 的 `error_message`。在 `IngestionEngine.executeNode()` 打断点，Step Into `ParserNode.execute()`，看原始 bytes 和 Tika 解析结果。常见原因：PDF 加密、图片 PDF 无 OCR、文件编码异常、文件损坏。

### 3.6 扩展改造（5 题）

**Q32：如果要在 Ragent 里接入一个新的向量数据库，比如 Weaviate，需要改哪些地方？**

> 需要实现 `VectorStoreService` 和 `RetrieverService` 接口，参考 `PgVectorStoreService` 和 `PgRetrieverService`。新增 `rag.vector.type=weaviate` 分支。如果 DO/Mapper 结构不同，可能还需要新增 Entity。最后修改 `application.yaml` 增加 Weaviate 连接配置。

**Q33：如果要给 RAG 回答增加引用来源，你会怎么设计？**

> 后端在 `RAGPromptService` 中让模型输出引用标记；`RetrievalContext` 中保留 chunk 的 `docId` 和 `chunkIndex`；`StreamChatEventHandler.onComplete()` 时把引用列表附加到消息。前端 `MessageItem` 渲染引用区域。数据库 `t_message` 可能新增 `references_json` 字段存储引用。MVP 版本可以只做后端解析和前端展示，不强制模型输出格式。

**Q34：如何优化 RAG 回答的 TTFT？**

> TTFT 由排队、记忆加载、问题改写、意图识别、检索、Rerank、模型首包组成。优化方向包括：缓存常见问题的改写结果；对低置信度意图跳过意图树遍历；检索通道做并行和缓存；Rerank 失败时快速 fallback 到 noop；模型候选按网络延迟动态排序。具体优化需要结合 Trace 数据分析瓶颈。

**Q35：如果要支持多租户，需要改哪些地方？**

> 数据库表增加 `tenant_id` 字段，并在 MyBatis 查询中统一加租户过滤。`UserContext` 增加租户信息。知识库、会话、文档等数据按租户隔离。MCP 工具调用可能也需要按租户路由。认证层面可以用 Sa-Token 的登录账号扩展字段。

**Q36：如何给 Ragent 增加一个自定义 MCP 工具？**

> 在 `mcp-server` 模块新增一个 Executor 类，参考 `WeatherMcpExecutor`，用 `@Component` + `@Bean` 返回 `SyncToolSpecification`。工具要有 `name`、`description` 和参数 schema。重启 MCP Server 后，`bootstrap` 的 `McpClientAutoConfiguration` 会自动发现。然后在管理后台意图树中配置对应 `mcp_tool_id`。

## 4. 面试官追问时怎么展开

### 4.1 通用展开技巧

1. **先给结论，再给证据**。
   - 差回答：“我们用了熔断。”
   - 好回答：“模型层用了类熔断器，状态机在 `ModelHealthStore` 里维护，失败阈值是 `ai.selection.failure-threshold=2`，打开持续时间是 `open-duration-ms=30000`。”

2. **用数字说话**。
   - “向量维度 1536”、“SSE 超时 5 分钟”、“历史保留 4 轮、摘要从第 5 轮开始”。

3. **区分“我确认过”和“我推测”**。
   - “源码中 `RagTraceRunDO` 只有 `RUNNING/SUCCESS/ERROR` 三种状态，这是我确认过的；用户取消是否会单独记录为 `CANCELLED`，我在源码中没有看到，需要进一步验证。”

4. **主动暴露边界条件**。
   - “通用 Ingestion 链路会写 `t_ingestion_task_node`，但知识库文档链路复用 `IngestionEngine` 时不会写节点表，这是源码里要注意的。”

### 4.2 常见追问及展开

**追问 1：你刚才说模型有熔断，具体怎么实现的？**

> 展开：`ModelHealthStore` 内部用 `ConcurrentHashMap<String, CircuitBreaker>` 维护每个候选的状态。`allowCall()` 检查状态，OPEN 时直接返回 false；HALF_OPEN 时允许少量探测；CLOSED 时正常调用。`markFailure()` 增加失败计数，达到 `failure-threshold` 就 OPEN，并记录打开时间。`markSuccess()` 在 HALF_OPEN 下连续成功则 CLOSED。

**追问 2：首包探测失败会怎样？**

> 展开：`RoutingLLMService.streamChat()` 遍历候选，对每个候选创建 `ProbeStreamBridge`。`LlmFirstPacketProbe.awaitFirstPacket(bridge, 60, SECONDS)` 等待首包。如果超时或首包失败，调用 `markFailure()` 并 cancel 当前流，继续下一个候选。全部失败抛 `RemoteException`，最终前端收到 `error` SSE 事件。

**追问 3：你们怎么保证 Trace 跨线程传播？**

> 展开：`RagTraceContext` 基于 `TransmittableThreadLocal`（TTL），它会在提交到线程池时做 `copy()`。`RagTraceNode` 的节点栈在 `copy()` 中做深拷贝，避免异步任务间父子节点 ID 混乱。`RagStreamTraceSupportImpl` 则在流式回调线程中手动管理节点生命周期。

**追问 4：检索为空时系统会怎么处理？**

> 展开：`StreamChatPipeline.handleEmptyRetrieval()` 判断 `RetrievalContext.isEmpty()`。如果是，根据配置可能直接返回“未检索到相关知识”的 system-only 回答，不走 LLM 生成。具体文案和分支可以在 `StreamChatPipeline` 中查看。

**追问 5：如果 MCP Server 挂了会怎样？**

> 展开：`bootstrap` 启动时会调 `tools/list`。如果 MCP Server 未启动，`registerRemoteTools()` 可能失败或工具列表为空。问答时如果意图命中 MCP 但工具未注册，参数提取或工具执行会失败。当前源码中没有看到强fallback，实际表现需要本地运行验证。
