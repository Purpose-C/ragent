# 面试复盘与项目讲解稿

## 1 分钟介绍

Ragent 是一个 Java 17 和 Spring Boot 3.5 实现的企业级 Agentic RAG 平台。它包含 React 管理端、PostgreSQL/pgvector 与 Milvus 双向量方案、Redis 分布式限流、RocketMQ、S3 对象存储，以及独立 MCP 工具服务。文档通过 Fetcher、Parser、Enhancer、Chunker、Enricher、Indexer Pipeline 入库；问答通过会话记忆、问题改写、意图识别、多路检索、去重 Rerank、Prompt 组装和 SSE 输出。模型层支持多供应商优先级路由和三态熔断。

## 3 分钟介绍框架

1. **业务目标**：让企业私有知识和实时业务工具统一服务于问答。
2. **架构分层**：`framework` 放通用能力，`infra-ai` 屏蔽模型差异，`bootstrap` 编排业务，`mcp-server` 独立提供工具。
3. **入库流程**：用户上传后创建任务，`IngestionEngine` 按数据库 Pipeline 执行六类节点，Embedding 后写 pgvector/Milvus，并保存节点级状态。
4. **问答流程**：`RAGChatController` 建 SSE，进入限流和 Trace；`StreamChatPipeline` 加载记忆、改写、识别意图、必要时引导澄清，再检索/MCP、组 Prompt、流式调用模型。
5. **可靠性**：模型候选优先级、三态熔断、首包探测、降级链；任务和 RAG 节点均有追踪记录。
6. **扩展性**：检索通道、后处理器、模型 Client 和 MCP 工具都采用接口/注册表方式扩展。

## 核心流程讲法

### 文档入库

入口为 `IngestionTaskController` 或知识库上传 Controller，Service 创建任务，`IngestionEngine` 读取 `PipelineDefinition`。Fetcher 获取文件，Parser 解析，Enhancer/Enricher 可用模型增强，Chunker 切块，Indexer 调 `EmbeddingService` 并写 `VectorStoreService`。任务和节点结果分别写 `t_ingestion_task`、`t_ingestion_task_node`。

### RAG 问答

入口是 `RAGChatController.chat()`。`RAGChatServiceImpl` 生成 taskId、进入分布式排队和 Trace。`StreamChatPipeline.execute()` 串联记忆、改写、意图、引导、检索和生成。`StreamChatEventHandler` 把 `meta/message/finish/done` 事件推给前端，并持久化回答。

## 可突出亮点

- 多通道召回后使用可插拔后处理器去重和 Rerank。
- 知识意图与 MCP 工具意图在统一 Context 中融合。
- 模型层不是硬编码单供应商，而是候选链、健康状态和流式首包探测。
- 入库任务和 RAG 请求都有节点级可观测性。
- 长会话同时使用近期消息和摘要，控制 Token 成本。

## 常见面试题

**问：为什么要 Rerank？** 召回阶段重覆盖率，会拿到较多候选；Rerank 用更强相关性模型重新排序，让送进 LLM 的少量上下文更准确。项目入口是 `RoutingRerankService`。

**问：模型挂了怎么办？** `ModelSelector` 给出候选，`ModelHealthStore` 用 CLOSED/OPEN/HALF_OPEN 维护健康，`ModelRoutingExecutor` 失败后尝试下一个。流式 Chat 还做首包探测。

**问：如何减少多轮对话 Token？** `DefaultConversationMemoryService` 并行加载近期历史和摘要，`JdbcConversationMemorySummaryService` 在达到阈值后压缩旧消息。

**问：MCP 和 RAG 如何选择？** 意图树节点包含知识库或 `mcp_tool_id`。知识问题走检索，实时业务问题走参数提取和工具调用，结果最终统一进入 Prompt。

**问：为何不直接用单一向量搜索？** 精确词、意图范围和语义召回各有优势。项目用意图定向与全局向量通道，再去重和重排。
