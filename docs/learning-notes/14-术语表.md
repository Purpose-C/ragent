# 术语表

> 本术语表不是百科，而是“Ragent 源码学习地图”。每个术语都指向源码中的具体位置，帮助你把概念和代码对应起来。

## RAG

**通俗解释**

RAG（Retrieval-Augmented Generation，检索增强生成）就是：用户提问后，系统先从知识库里找出相关内容，再把内容塞进 Prompt 让大模型回答。这样模型回答时就能基于企业私有知识，而不是只靠训练时学到的内容。

**在 Ragent 中的具体位置**

- 核心编排：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/service/pipeline/StreamChatPipeline.java`
- 检索阶段：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/retrieve/RetrievalEngine.java`
- Prompt 组装：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/prompt/RAGPromptService.java`

**相关类 / 方法 / 配置 / 表**

- `StreamChatPipeline.execute()`
- `RetrievalEngine.retrieve()`
- `RAGPromptService.buildStructuredMessages()`
- `resources/prompt/answer-chat-kb.st`
- `t_knowledge_vector`、`t_message`

**容易误解的点**

RAG 不是“先百度一下再回答”。它是结构化的：问题改写 → 意图识别 → 检索 → 去重 → Rerank → Prompt 组装 → 模型生成，每一步都有明确代码。

**一个学习问题**

为什么 Ragent 不直接把用户问题丢给大模型，而是要先做检索？

## Agentic RAG

**通俗解释**

Agentic RAG 是普通 RAG 的升级版。普通 RAG 只会“检索 + 生成”；Agentic RAG 会在中间做决策：是否需要改写问题？走知识库还是调工具？是否需要澄清？是否需要多步推理？

**在 Ragent 中的具体位置**

- 问题改写：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/rewrite/MultiQuestionRewriteService.java`
- 意图识别：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/intent/IntentResolver.java`
- 工具调用：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/mcp/McpClientToolExecutor.java`
- 主编排：`StreamChatPipeline.java`

**相关类 / 方法 / 配置 / 表**

- `MultiQuestionRewriteService.rewriteWithSplit()`
- `IntentResolver.resolve()`
- `McpClientToolExecutor.execute()`
- `t_intent_tree`、`t_intent_node`

**容易误解的点**

Agentic RAG 不等于“让模型无限自主循环”。Ragent 的决策是有限且可控的：改写、意图、澄清、检索/MCP，每个分支在 `StreamChatPipeline` 中明确写出。

**一个学习问题**

Ragent 中哪些步骤体现了“Agentic”能力？

## MCP

**通俗解释**

MCP（Model Context Protocol）是一个标准协议，让 AI 应用用统一方式发现、调用外部工具。比如天气查询、工单查询、销售数据查询，都可以封装成 MCP 工具，Ragent 通过 MCP 客户端统一调用。

**在 Ragent 中的具体位置**

- MCP Server：`mcp-server/src/main/java/com/nageoffer/ai/ragent/mcp/McpServerApplication.java`
- 工具实现：`mcp-server/src/main/java/com/nageoffer/ai/ragent/mcp/executor/WeatherMcpExecutor.java`
- 客户端注册：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/mcp/McpClientAutoConfiguration.java`
- 工具执行：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/mcp/McpClientToolExecutor.java`

**相关类 / 方法 / 配置 / 表**

- `WeatherMcpExecutor`、`TicketMcpExecutor`、`SalesMcpExecutor`
- `DefaultMcpToolRegistry`
- `rag.mcp.servers`（`application.yaml`）
- `t_intent_node.mcp_tool_id`

**容易误解的点**

MCP 不是向量数据库，也不是知识库。它解决的是“实时业务数据”和“外部系统能力”的调用问题，与 RAG 检索互补。

**一个学习问题**

用户问“北京今天天气怎么样”和“公司请假制度是什么”，Ragent 分别会走哪两条不同的路径？

## Embedding

**通俗解释**

Embedding 是把一段文本变成一串数字向量。语义相近的文本，向量在数学空间中也更接近。Ragent 用 Embedding 实现“按意思找内容”，而不是只能按关键词匹配。

**在 Ragent 中的具体位置**

- 能力接口：`infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/embedding/EmbeddingService.java`
- 路由实现：`infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/embedding/RoutingEmbeddingService.java`
- 入库调用：`bootstrap/src/main/java/com/nageoffer/ai/ragent/ingestion/service/ChunkEmbeddingService.java`

**相关类 / 方法 / 配置 / 表**

- `EmbeddingService.embed()` / `embedBatch()`
- `RoutingEmbeddingService.embedBatch()`
- `ai.embedding.candidates`（`application.yaml`）
- `rag.default.dimension=1536`（`application.yaml`）
- `t_knowledge_vector.embedding`

**容易误解的点**

Embedding 不是生成答案，它只是把文本转成向量。同一个 Chunk 的文本给模型读，Embedding 给向量检索用。

**一个学习问题**

为什么 `t_knowledge_vector.embedding` 的维度必须与 `rag.default.dimension` 一致？

## Rerank

**通俗解释**

Rerank 是对召回结果再次排序。检索阶段为了覆盖率可能拿回很多候选，Rerank 用更强的相关性模型判断哪些候选真正相关，把最相关的排在前面，再取 Top-K 给大模型。

**在 Ragent 中的具体位置**

- 能力接口：`infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/rerank/RerankService.java`
- 路由实现：`infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/rerank/RoutingRerankService.java`
- 后处理器：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/retrieve/postprocessor/RerankPostProcessor.java`

**相关类 / 方法 / 配置 / 表**

- `RerankService.rerank()`
- `RoutingRerankService.rerank()`
- `RerankPostProcessor.process()`
- `ai.rerank.candidates`（`application.yaml`）
- `rag.rerank.enabled=true`

**容易误解的点**

Rerank 不负责“首次召回”。如果检索阶段什么都没召回，Rerank 也无能为力。

**一个学习问题**

Ragent 中 Rerank 失败后会不会导致整个问答失败？为什么？

## Vector Store（向量存储）

**通俗解释**

向量存储是专门保存和检索向量的数据库或扩展。Ragent 把 Chunk 的 Embedding 存进去，用户提问时再把问题向量也存进去做相似度搜索。

**在 Ragent 中的具体位置**

- pgvector 实现：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/vector/PgVectorStoreService.java`
- Milvus 实现：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/vector/MilvusVectorStoreService.java`
- 检索实现：`PgRetrieverService` / `MilvusRetrieverService`

**相关类 / 方法 / 配置 / 表**

- `VectorStoreService.save()` / `delete()`
- `RetrieverService.retrieve()`
- `rag.vector.type=pg`（`application.yaml`）
- `t_knowledge_vector` 或 Milvus Collection

**容易误解的点**

向量存储不只存向量，还存 Chunk 正文和 metadata。metadata 里记录了 `knowledgeBaseId`、`docId`、`chunkIndex`，用于关联业务数据。

**一个学习问题**

`rag.vector.type=pg` 和 `rag.vector.type=milvus` 切换时，哪些表会变、哪些表不变？

## pgvector

**通俗解释**

pgvector 是 PostgreSQL 的一个扩展，让普通 PostgreSQL 数据库也能存向量并做相似度检索。Ragent 默认用它作为向量存储方案。

**在 Ragent 中的具体位置**

- 建表 SQL：`resources/database/schema_pg.sql`
- 向量字段：`t_knowledge_vector.embedding vector(1536)`
- 索引：`CREATE INDEX idx_kv_embedding ON t_knowledge_vector USING hnsw (embedding vector_cosine_ops);`

**相关类 / 方法 / 配置 / 表**

- `PgVectorStoreService`
- `PgRetrieverService`
- `t_knowledge_vector`
- `rag.default.metric-type=COSINE`

**容易误解的点**

pgvector 不是独立数据库进程，它是 PostgreSQL 的扩展。你仍然连同一个 PG 实例，只是多了一张带向量字段的表。

**一个学习问题**

pgvector 和 Milvus 在 Ragent 中的共同职责是什么？差异是什么？

## Milvus

**通俗解释**

Milvus 是一个独立的向量数据库，专门做高并发、大规模的向量检索。Ragent 把它作为 pgvector 的可选替代方案。

**在 Ragent 中的具体位置**

- 配置：`application.yaml` 中 `milvus.uri=http://localhost:19530`
- 实现：`MilvusVectorStoreService`、`MilvusRetrieverService`

**相关类 / 方法 / 配置 / 表**

- `MilvusVectorStoreService`

- `rag.vector.type=milvus`

**容易误解的点**

配置了 Milvus 不代表可以不要 PostgreSQL。用户、会话、文档、Chunk、任务、Trace 等结构化数据仍然依赖 PostgreSQL。

## Chunk

**通俗解释**

Chunk 是把长文档切分成的小段文本。Ragent 检索和生成时，给模型看的不是整篇文档，而是最相关的几个 Chunk。

**在 Ragent 中的具体位置**

- 切块节点：`bootstrap/src/main/java/com/nageoffer/ai/ragent/ingestion/node/ChunkerNode.java`
- 业务表：`bootstrap/src/main/java/com/nageoffer/ai/ragent/knowledge/dao/entity/KnowledgeChunkDO.java`

**相关类 / 方法 / 配置 / 表**

- `ChunkerNode.execute()`
- `KnowledgeChunkDO`
- `t_knowledge_chunk`
- `t_knowledge_vector`（每个 Chunk 对应一条向量记录）

**容易误解的点**

Chunk 不是越小越好。太小会丢失上下文，太大会降低检索精度。项目通过 `chunkSize` 和 `overlapSize` 控制。

**一个学习问题**

为什么 Ragent 需要同时保存 `t_knowledge_chunk` 和 `t_knowledge_vector`？

## Top-K

**通俗解释**

Top-K 就是“取最相关的前 K 条”。检索阶段召回很多候选，但最终只把最相关的 K 个 Chunk 塞进 Prompt，避免上下文过长。

**在 Ragent 中的具体位置**

- 检索配置：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/config/SearchChannelProperties.java`
- 多路召回：`MultiChannelRetrievalEngine`

**相关类 / 方法 / 配置 / 表**

- `SearchChannelProperties`
- `rag.search.channels.vector-global.top-k-multiplier=3`
- `rag.search.channels.intent-directed.top-k-multiplier=2`

**容易误解的点**

K 不是越大越好。K 越大，给模型的上下文越长，成本和噪声都越高。Ragent 用 multiplier 在候选阶段放宽，再经 Rerank 收紧。

**一个学习问题**

`top-k-multiplier` 和最终 Top-K 是什么关系？

## Prompt

**通俗解释**

Prompt 是给大模型的指令和上下文。Ragent 的 Prompt 通常包含：系统角色说明、检索到的知识、历史对话、当前问题。

**在 Ragent 中的具体位置**

- `resources/prompt/answer-chat-mcp.st`
- `framework/src/main/java/com/nageoffer/ai/ragent/framework/convention/ChatRequest.java`

**容易误解的点**

Prompt 不只是“用户问题”。在 Ragent 中，Prompt 是结构化消息列表，包含 system、user、assistant 多个角色。

## Token

**通俗解释**

Token 是大模型处理文本的基本单位。一个中文词可能对应 1～2 个 Token，一个英文单词可能对应 1 个 Token。Token 数量决定成本、上下文窗口占用和响应速度。

**在 Ragent 中的具体位置**

- 记忆配置：`application.yaml` 中 `rag.memory.*`
- 摘要服务：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/memory/JdbcConversationMemorySummaryService.java`

**相关类 / 方法 / 配置 / 表**

- `rag.memory.history-keep-turns=4`
- `rag.memory.summary-start-turns=5`
- `rag.memory.summary-max-chars=200`
- `t_conversation_summary`

**容易误解的点**

Token 数不等于字符数。Ragent 目前用字符数/轮次做近似控制，没有精确 tokenizer。

**一个学习问题**

为什么长会话要引入摘要机制？

## SSE

**通俗解释**

SSE（Server-Sent Events）是服务端向浏览器单向推送事件的技术。Ragent 用 SSE 把大模型生成的文字逐字推给前端，让用户不用等整个回答完成。

**在 Ragent 中的具体位置**

- 后端：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/controller/RAGChatController.java` 返回 `SseEmitter`
- 前端：`frontend/src/hooks/useStreamResponse.ts`
- 事件处理：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/service/handler/StreamChatEventHandler.java`

**相关类 / 方法 / 配置 / 表**

- `RAGChatController.chat()`
- `useStreamResponse.createStreamResponse()` / `readSseStream()`
- `StreamChatEventHandler.onMeta()` / `onMessage()` / `onFinish()` / `onComplete()`
- `rag.default.sse-timeout-ms=300000`

**容易误解的点**

SSE 不是 WebSocket。SSE 是单向的、基于 HTTP 的，适合模型推流；WebSocket 是双向全双工的。

**一个学习问题**

Ragent 为什么用 fetch 而不是原生 `EventSource` 读取 SSE？

## JSON-RPC

**通俗解释**

JSON-RPC 是一种用 JSON 格式进行远程调用的协议。MCP 底层使用 JSON-RPC 消息格式，`tools/list` 和 `callTool` 都是 JSON-RPC 方法。

**在 Ragent 中的具体位置**

- MCP SDK 内部封装，项目没有手写完整协议栈。
- 可参考 `mcp-server` 中 MCP SDK 的 `SyncToolSpecification` 和 `callTool()` 调用。

**相关类 / 方法 / 配置 / 表**

- `mcp-server/pom.xml` 中 `io.modelcontextprotocol.sdk:mcp`
- `WeatherMcpExecutor` 返回的 `CallToolResult`

**容易误解的点**

Ragent 没有从零实现 JSON-RPC，而是依赖 MCP Java SDK。理解到“请求/响应格式是 JSON-RPC”即可，不需要手写协议解析。

**一个学习问题**

MCP 工具发现和工具调用分别对应 JSON-RPC 的哪两个方法？

## 模型路由

**通俗解释**

模型路由就是：业务代码只说“我要调大模型”，具体调哪个供应商、哪个模型，由路由层根据候选列表、健康状态、优先级来决定。

**在 Ragent 中的具体位置**

- 能力接口：`infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/LLMService.java`
- 路由实现：`infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/RoutingLLMService.java`
- 候选选择：`infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/ModelSelector.java`

**相关类 / 方法 / 配置 / 表**

- `LLMService.chat()` / `streamChat()`
- `RoutingLLMService.chat()` / `streamChat()`
- `ModelSelector.selectChatCandidates()`
- `ai.chat.candidates`（`application.yaml`）

**容易误解的点**

模型路由不是随机负载均衡。它根据 `priority`、`supports-thinking`、健康状态、是否指定 `modelId` 等多维度选择。

**一个学习问题**

业务代码注入 `LLMService` 时，实际拿到的是哪个实现类？

## 熔断

**通俗解释**

熔断就是：当一个模型持续失败时，暂时不再调用它，避免把更多请求打到已经挂掉的服务上。等过一段时间再尝试恢复。

**在 Ragent 中的具体位置**

- 健康状态：`infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/ModelHealthStore.java`

**相关类 / 方法 / 配置 / 表**

- `ModelHealthStore.allowCall()`
- `ModelHealthStore.markFailure()` / `markSuccess()`
- `ai.selection.failure-threshold=2`
- `ai.selection.open-duration-ms=30000`

**容易误解的点**

熔断不是永久禁用。状态会在 CLOSED（正常）、OPEN（断开）、HALF_OPEN（试探）之间转换。

**一个学习问题**

`failure-threshold=2` 和 `open-duration-ms=30000` 组合起来是什么含义？

## 降级

**通俗解释**

降级就是：主候选失败后，自动尝试下一个候选。虽然质量可能下降，但至少保证服务可用。

**在 Ragent 中的具体位置**

- 同步降级：`infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/ModelRoutingExecutor.java`
- 流式降级：`RoutingLLMService.streamChat()` 内部循环

**相关类 / 方法 / 配置 / 表**

- `ModelRoutingExecutor.executeWithFallback()`
- `RoutingLLMService.streamChat()`
- `ai.chat.candidates` 中的 `priority`

**容易误解的点**

降级后的模型质量可能下降。比如从 qwen-max 降到 noop Rerank，排序质量会变差，这是设计上的 trade-off。

**一个学习问题**

同步调用和流式调用在降级逻辑上有什么区别？

## 首包探测

**通俗解释**

首包探测就是：在把模型第一个字推给用户之前，先判断当前模型是否能正常开始输出。如果首包失败，可以安全切换到下一个候选，避免用户看到半截回答。

**在 Ragent 中的具体位置**

- 流式路由：`RoutingLLMService.streamChat()`
- 探测桥接：`infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/ProbeStreamBridge.java`

**相关类 / 方法 / 配置 / 表**

- `ProbeStreamBridge`
- `LlmFirstPacketProbe.awaitFirstPacket()`
- `RoutingLLMService` 中 `FIRST_PACKET_TIMEOUT_SECONDS = 60`

**容易误解的点**

首包探测只在“首字发出前”有效。一旦首字发出，就不能再切换候选，否则用户会看到重复开头。

**一个学习问题**

为什么流式调用不能简单重试，而要做首包探测？

## 会话记忆

**通俗解释**

会话记忆就是让大模型能“记得”之前的对话。Ragent 会保存近期消息，并在轮次较多时生成摘要，避免上下文过长。

**在 Ragent 中的具体位置**

- 记忆服务：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/memory/DefaultConversationMemoryService.java`
- 摘要服务：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/memory/JdbcConversationMemorySummaryService.java`

**相关类 / 方法 / 配置 / 表**

- `DefaultConversationMemoryService.load()` / `append()`
- `JdbcConversationMemorySummaryService`
- `rag.memory.history-keep-turns=4`
- `rag.memory.summary-start-turns=5`
- `t_message`、`t_conversation_summary`

**容易误解的点**

会话记忆不是无限保存历史。超过 `history-keep-turns` 的旧消息会被摘要替代，控制 Token 成本。

**一个学习问题**

`history-keep-turns=4` 和 `summary-start-turns=5` 配合起来是什么意思？

## 意图识别

**通俗解释**

意图识别就是：判断用户问题属于哪一类。Ragent 中主要有三类意图：知识库（KB）、系统（SYSTEM）、MCP 工具（MCP）。不同意图走不同处理路径。

**在 Ragent 中的具体位置**

- 意图解析：`bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/intent/IntentResolver.java`
- 意图树配置：`t_intent_tree`、`t_intent_node`

**相关类 / 方法 / 配置 / 表**

- `IntentResolver.resolve()`
- `IntentKind`（KB / SYSTEM / MCP）
- `t_intent_tree`、`t_intent_node`

**容易误解的点**

意图识别不是关键词 if-else。Ragent 用模型对问题做分类，再结合意图树节点配置做决策。

**一个学习问题**

一个用户问题可能同时命中 KB 和 MCP 意图吗？如果会，Ragent 怎么处理？

## 问题重写

## 多路召回

**通俗解释**

多路召回就是：用多个不同策略同时找相关 Chunk。比如一路按意图定向检索，一路跨所有知识库全局检索，最后把结果融合。

## Trace

**容易误解的点**

## 幂等

**通俗解释**

幂等就是：同样的请求执行多次，结果和副作用与执行一次相同。Ragent 用幂等防止用户重复点击提交、防止 MQ 重复消费。

**在 Ragent 中的具体位置**

- 幂等注解：`framework/src/main/java/com/nageoffer/ai/ragent/framework/idempotent/IdempotentSubmit.java`
- 幂等切面：`framework/src/main/java/com/nageoffer/ai/ragent/framework/idempotent/IdempotentSubmitAspect.java`

**相关类 / 方法 / 配置 / 表**

- `@IdempotentSubmit`
- `IdempotentSubmitAspect.around()`
- `RAGChatController.chat()` 上的 `@IdempotentSubmit`
- Redisson 分布式锁

**容易误解的点**

幂等不是“请求只执行一次”。它允许请求被处理多次，但保证副作用只发生一次。

**一个学习问题**

## 本章复习问题

1. RAG 和 Agentic RAG 的主要区别是什么？
2. Embedding 和 Rerank 在流程中分别承担什么职责？
3. MCP 和知识库检索分别解决什么问题？
4. `pgvector` 和 `Milvus` 在 Ragent 中是什么关系？
5. `Chunk` 和 `Top-K` 分别是什么？
6. SSE 和 WebSocket 的主要区别是什么？
7. 模型路由、熔断、降级、首包探测分别解决什么问题？
8. `Pipeline` 和 `Node` 是什么关系？
9. `DTO`、`DO`、`Mapper` 分别用于哪一层？
10. `Trace` 和普通日志有什么区别？

## 参考答案

1. RAG 只做检索 + 生成；Agentic RAG 增加了问题改写、意图识别、工具调用等决策能力。
2. Embedding 把文本转成向量用于检索；Rerank 对召回候选重新排序。
3. MCP 解决实时业务工具调用；知识库检索解决静态文档内容召回。
4. 二者是可替换的向量存储后端，由 `rag.vector.type` 选择；但结构化业务数据仍依赖 PostgreSQL。
5. Chunk 是文档切分后的小段文本；Top-K 是取最相关的前 K 个 Chunk。
6. SSE 是服务端单向推流，基于 HTTP；WebSocket 是双向全双工。
7. 模型路由选候选；熔断暂停调用失败模型；降级切到备选模型；首包探测避免用户看到半截回答。
8. Pipeline 是节点链，Node 是单个处理步骤。
9. DTO 用于接口层传参/响应；DO 映射数据库表；Mapper 做数据库 CRUD。
10. Trace 是结构化的请求级瀑布图，可跨线程传播和可视化；普通日志是文本化运行轨迹。
