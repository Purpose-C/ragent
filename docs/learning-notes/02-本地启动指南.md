# 本地启动指南

> 本章目标：把 Ragent 从零 clone 到能登录、能聊天、能入库，写成一本“手摸手”操作手册。即使你之前没搭过 PostgreSQL、没用过 RocketMQ、没调过大模型 API，也能按步骤跑起来。

---

## 1. 启动前总览

### 1.1 四个角色

Ragent 本地启动涉及四个角色：前端、后端、MCP 服务、中间件。

| 角色 | 是什么 | 默认端口 | 是否必须启动 |
|---|---|---|---|
| **前端（frontend）** | React 18 + Vite 写的用户界面和管理后台 | 5173 | 是（要看页面） |
| **后端（bootstrap）** | Spring Boot 主业务服务 | 9090 | 是 |
| **MCP 服务（mcp-server）** | 独立 Spring Boot 工具服务，暴露天气/工单/销售工具 | 9099 | 否（不验证 MCP 时可后启） |
| **中间件** | PostgreSQL、Redis、RocketMQ、RustFS、Milvus | 见下表 | PostgreSQL + Redis 必须；其余按需 |

### 1.2 中间件默认端口

| 中间件 | 默认端口 | 作用 | 是否必须 |
|---|---|---|---|
| PostgreSQL | 5432 | 存用户、会话、消息、知识库、文档、Trace 等结构化数据 | **必须** |
| Redis | 6379 | Sa-Token 会话、限流、幂等、取消信号广播 | **必须** |
| RocketMQ NameServer | 9876 | 消息队列名字服务，知识库文档分块异步执行 | 可选（不触发分块可先不启） |
| RocketMQ Broker | 10911 / 10909 / 10912 | 消息队列 Broker | 可选 |
| RustFS / S3 | 9000（API）/ 9001（Console） | 对象存储，保存上传的源文件 | 可选（知识库上传需要） |
| Milvus | 19530（gRPC）/ 9091（HTTP） | 可选向量数据库 | 可选（默认用 pgvector） |

### 1.3 最小启动组合

如果你只想先看界面和简单问答，最小组合是：

```text
PostgreSQL + Redis + 后端 + 前端
```

此时：

- 可以登录；
- 可以聊天（但模型需要 API Key）；
- 不能上传文档（需要 RustFS/RocketMQ）；
- 不能用 MCP 工具（需要 mcp-server）。

**推荐学习顺序**：

```text
第 1 步：PostgreSQL + Redis
第 2 步：后端 bootstrap
第 3 步：前端 frontend
第 4 步：模型 API Key（才能正常问答）
第 5 步：RocketMQ + RustFS（才能文档入库）
第 6 步：MCP Server（才能工具调用）
第 7 步：Milvus（可选，默认 pgvector 足够学习）
```

## 2. 环境准备

### 2.1 JDK 17

**为什么需要**：根 `pom.xml` 指定 Java 17，Spring Boot 3.5 也要求 JDK 17+。

**检查命令**：

```bash
java -version
```

期望看到类似输出：

```text
openjdk version "17.0.x" 20xx-xx-xx
OpenJDK Runtime Environment (build 17.0.x+xx)
```

**如果没有 JDK 17**：

- macOS：`brew install openjdk@17`
- Windows/Linux：从 [Adoptium](https://adoptium.net/) 下载 JDK 17。

配置环境变量（示例，macOS/Linux）：

```bash
export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-17.jdk/Contents/Home
export PATH=$JAVA_HOME/bin:$PATH
```

**验证**：

```bash
java -version
javac -version
```

### 2.2 Maven Wrapper 和系统 Maven

项目根目录有 `./mvnw`（Maven Wrapper），它会自动下载项目指定的 Maven 版本，不需要你本地先装 Maven。

**为什么用 Wrapper**：

- 保证所有人用同一 Maven 版本，避免“我本地能跑你不行”。
- 避免手动配置镜像和插件仓库。

**检查 Wrapper 是否可用**：

```bash
./mvnw -v
```

**如果你本地已有 Maven**，也可以用：

```bash
mvn -v
```

但建议优先用 `./mvnw`，除非网络极差需要走本地 Maven 缓存。

### 2.3 Node / npm

**为什么需要**：前端用 Vite 构建，依赖 npm 包。

**检查命令**：

```bash
node -v
npm -v
```

`frontend/package.json` 没有显式声明最低 Node 版本，但依赖的 Vite 5 建议 Node 18+。推荐使用 Node 18 LTS 或 20 LTS。

**如果没有 Node**：

- 推荐用 [nvm](https://github.com/nvm-sh/nvm) 安装：

```bash
nvm install 20
nvm use 20
```

### 2.4 Docker 是否必须

**不是必须，但强烈推荐**。

Ragent 的中间件（PostgreSQL、Redis、RocketMQ、RustFS、Milvus）都可以：

- **用 Docker 一键启动**（推荐，省时间）；
- **或手工安装**（适合想深入理解每个中间件的人）。

**初学建议**：

- 如果你电脑有 Docker，直接用仓库提供的 Compose 文件。
- 如果你没有 Docker，可以先手工装 PostgreSQL + Redis，其余按需再装。

### 2.5 中间件角色

| 中间件 | 角色 | Ragent 中具体用途 |
|---|---|---|
| **PostgreSQL** | 主业务数据库 | 用户、会话、消息、知识库、文档、Chunk、Pipeline、Trace 等 |
| **pgvector** | PostgreSQL 扩展 | 存 Chunk 的 Embedding 向量，做相似度检索 |
| **Redis** | 缓存 / 会话 / 分布式锁 | Sa-Token Token 存储、聊天限流排队、幂等锁、SSE 取消信号广播 |
| **RocketMQ** | 异步消息队列 | 知识库文档上传后异步触发分块任务 |
| **RustFS** | S3 兼容对象存储 | 保存上传的 PDF、Word、TXT 源文件 |
| **Milvus** | 可选向量数据库 | 替代 pgvector 做大规模向量检索 |

**初学者注意**：

- 默认 `rag.vector.type=pg`，所以 Milvus 可以暂时不启动。
- RustFS 只有上传知识库文档时才需要。
- RocketMQ 只有触发知识库“分块”时才需要。

## 3. 克隆仓库与分支建议

### 3.1 Fork 后 clone

**为什么不直接 clone 原仓库**：

- 你会在本地写学习笔记、做实验，直接 clone 原仓库不方便提交自己的改动。
- Fork 后你的仓库是 `origin`，原仓库可以设为 `upstream`，便于同步更新。

**操作步骤**：

1. 在 GitHub 上 Fork `nageoffer/ragent` 到你自己的账号。
2. 把你的 Fork clone 到本地：

```bash
git clone git@github.com:你的用户名/ragent.git
cd ragent
```

3. 添加原仓库为 upstream：

```bash
git remote add upstream git@github.com:nageoffer/ragent.git
```

4. 查看远程仓库：

```bash
git remote -v
```

期望看到：

```text
origin    git@github.com:你的用户名/ragent.git (fetch)
origin    git@github.com:你的用户名/ragent.git (push)
upstream  git@github.com:nageoffer/ragent.git (fetch)
upstream  git@github.com:nageoffer/ragent.git (push)
```

### 3.2 origin 和 upstream 的区别

- `origin`：你的 Fork，你可以 push 自己的学习分支和代码。
- `upstream`：原作者仓库，只用来拉取更新，通常不直接 push。

### 3.3 不要直接在 main 上实验

**为什么**：

- `main` 应该尽量保持干净，方便同步 upstream 更新。
- 实验代码如果混进 `main`，以后同步原项目时容易产生冲突。

**建议分支命名**：

```text
git checkout -b learn/local-startup-notes
```

或者：

```text
git checkout -b feat/my-first-experiment
```

## 4. 后端配置详解

Ragent 的主要配置文件是：

```text
bootstrap/src/main/resources/application.yaml
```

**为什么是它**：

- 这是 Spring Boot 默认加载的配置文件。
- 根目录没有 `application.yml`，不要把根 `resources/` 误当运行时资源目录。

### 4.1 逐段解释

#### server

```yaml
server:
  port: 9090
  servlet:
    context-path: /api/ragent
```

- `port: 9090`：后端服务端口。
- `context-path: /api/ragent`：所有接口前缀。前端请求的 `/api/xxx` 会被代理到 `http://localhost:9090/api/ragent/api/xxx`？不，Vite 代理的是 `/api` → `http://localhost:9090`，所以完整路径是 `http://localhost:9090/api/ragent/xxx`。

**初学阶段**：一般不用改。

#### spring.datasource

```yaml
spring:
  datasource:
    driver-class-name: org.postgresql.Driver
    type: com.zaxxer.hikari.HikariDataSource
    username: postgres
    password: postgres
    url: jdbc:postgresql://127.0.0.1:5432/ragent?client_encoding=UTF8
```

- 必须改：`username`、`password`、`url` 中的数据库名和地址。
- 初学阶段必须改，改成你自己的 PostgreSQL 账号。

#### spring.data.redis

```yaml
spring:
  data:
    redis:
      host: 127.0.0.1
      port: 6379
      password: 123456
```

- 必须改：`password`，如果本地 Redis 没有密码则留空或删除该行。
- `host` / `port` 如果 Redis 在本地默认端口，可不改。

#### rocketmq

```yaml
rocketmq:
  name-server: 127.0.0.1:9876
  producer:
    group: ragent-producer${unique-name:}_pg
    send-message-timeout: 2000
```

- 必须改：如果 RocketMQ 不在本地或端口不同，改 `name-server`。
- 初学阶段如果暂时不用知识库分块，可以先不改，但启动时可能会报错连不上 RocketMQ（可在日志中确认是否为致命错误）。

#### rag.vector

```yaml
rag:
  vector:
    type: pg  # 可选 milvus / pg
```

- `type=pg`：用 PostgreSQL pgvector。
- `type=milvus`：用 Milvus。
- 初学阶段建议保持 `pg`。

#### rag.default

```yaml
rag:
  default:
    collection-name: rag_default_store
    dimension: 1536
    metric-type: COSINE
    sse-timeout-ms: 300000
```

- `dimension: 1536`：向量维度，必须和 Embedding 模型输出一致。
- `metric-type: COSINE`：相似度度量，一般不用改。
- `sse-timeout-ms: 300000`：SSE 5 分钟超时。
- 初学阶段一般不用改，但要确认 Embedding 模型输出维度。

#### rag.query-rewrite / rag.rerank / rag.rate-limit / rag.memory

```yaml
rag:
  query-rewrite:
    enabled: true
  rerank:
    enabled: true
  rate-limit:
    global:
      enabled: true
      max-concurrent: 10
  memory:
    history-keep-turns: 4
    summary-start-turns: 5
```

- 这些控制 RAG 行为开关。
- 初学阶段一般不用改。如果模型 API Key 没配，可以先把 `rerank.enabled` 设为 `false`，避免 Rerank 失败影响问答。

#### rag.mcp

```yaml
rag:
  mcp:
    servers:
      - name: default
        url: http://localhost:9099
```

- 指向 MCP Server 地址。
- 初学阶段如果 MCP Server 没启动，可以暂时不改，但 MCP 工具无法使用。

#### ai.providers

```yaml
ai:
  providers:
    ollama:
      url: http://localhost:11434
    bailian:
      url: https://dashscope.aliyuncs.com
      api-key: ${BAILIAN_API_KEY:}
    aihubmix:
      url: https://aihubmix.com
      api-key: ${AIHUBMIX_API_KEY:}
    siliconflow:
      url: https://api.siliconflow.cn
      api-key: ${SILICONFLOW_API_KEY:}
```

- 配置各模型供应商的地址和 API Key。
- 必须改：把 `${BAILIAN_API_KEY:}` 等替换为真实 Key，或设置对应环境变量。
- `ollama` 不需要 API Key，但需要本地启动 Ollama 服务。

#### ai.chat / ai.embedding / ai.rerank

```yaml
ai:
  chat:
    default-model: qwen3-max
    candidates:
      - id: qwen-plus
        provider: bailian
        model: qwen-plus-latest
        priority: 1
```

- 配置 Chat、Embedding、Rerank 的候选模型。
- 初学阶段可以不改，但要确保候选模型对应的供应商 Key 已配置。

#### rustfs

```yaml
rustfs:
  url: http://localhost:9000
  access-key-id: rustfsadmin
  secret-access-key: rustfsadmin
```

- 对象存储配置。
- 初学阶段如果启动 RustFS Docker，可不改；如果用自己的 MinIO，需要改地址和密钥。

#### sa-token

```yaml
sa-token:
  token-name: Authorization
  timeout: 2592000
  is-concurrent: true
  is-share: false
  token-style: simple-uuid
```

- 认证框架配置。
- 初学阶段一般不用改。

### 4.2 初学阶段必须改的配置

| 配置段 | 必须改的字段 | 原因 |
|---|---|---|
| `spring.datasource` | `url`、`username`、`password` | 连你自己的 PostgreSQL |
| `spring.data.redis` | `password` | 连你自己的 Redis |
| `ai.providers.*` | `api-key` | 调用外部大模型 |

### 4.3 初学阶段暂时不用改的配置

| 配置段 | 原因 |
|---|---|
| `server.port` / `context-path` | 本地默认即可 |
| `rag.default.dimension` | 默认 1536，与多数 Embedding 模型一致 |
| `rag.vector.type` | 默认 pg，无需 Milvus |
| `sa-token.*` | 本地默认即可 |
| `rag.rate-limit.*` | 本地单人使用不会触发限流 |

## 5. 数据库初始化

### 5.1 创建 PostgreSQL 数据库

**为什么**：Ragent 用 PostgreSQL 存所有结构化数据。

**操作步骤**：

1. 用 psql 或 DBeaver 连接你的 PostgreSQL 实例。
2. 创建数据库：

```sql
CREATE DATABASE ragent
    WITH
    OWNER = postgres
    ENCODING = 'UTF8'
    LC_COLLATE = 'en_US.UTF-8'
    LC_CTYPE = 'en_US.UTF-8'
    TABLESPACE = pg_default
    CONNECTION LIMIT = -1;
```

3. 进入数据库：

```bash
psql -U postgres -d ragent
```

### 5.2 安装 pgvector 扩展

**为什么**：向量字段 `embedding vector(1536)` 需要 pgvector 扩展。

```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

**验证**：

```sql
SELECT * FROM pg_extension WHERE extname = 'vector';
```

如果看到 `vector` 行，说明安装成功。

### 5.3 执行 schema 和 init data

**为什么**：`schema_pg.sql` 创建表结构，`init_data_pg.sql` 插入初始管理员。

**执行顺序**：

```bash
# 1. 执行 schema（建表、索引、注释）
psql -U postgres -d ragent -f resources/database/schema_pg.sql

# 2. 执行 init data（初始管理员 admin/admin）
psql -U postgres -d ragent -f resources/database/init_data_pg.sql
```

**注意**：

- `schema_pg.sql` 是**最新完整结构**，不是增量脚本。
- 如果执行报错说表已存在，说明你可能重复执行了。

### 5.4 升级脚本什么时候不能执行

仓库中有：

```text
resources/database/upgrade_v1.0_to_v1.1.sql
resources/database/upgrade_v1.1_to_v1.2.sql
```

**什么时候不能执行**：

- 新库不要执行升级脚本，因为 `schema_pg.sql` 已经包含最新结构。
- 重复执行升级脚本会报错（字段已存在或已改名）。

**什么时候执行**：

- 只有从旧版本升级且需要保留数据时才按版本顺序执行。

### 5.5 初始化后应该能看到哪些表

执行完后，查表数量：

```sql
SELECT table_name
FROM information_schema.tables
WHERE table_schema = 'public'
  AND table_name LIKE 't_%'
ORDER BY table_name;
```

应该能看到约 21 张表，包括：

- `t_user`
- `t_conversation`、`t_message`、`t_conversation_summary`
- `t_knowledge_base`、`t_knowledge_document`、`t_knowledge_chunk`
- `t_knowledge_vector`
- `t_ingestion_pipeline`、`t_ingestion_pipeline_node`、`t_ingestion_task`、`t_ingestion_task_node`
- `t_rag_trace_run`、`t_rag_trace_node`
- `t_intent_tree`、`t_intent_node`
- 其他系统表

验证初始用户：

```sql
SELECT id, username, role FROM t_user;
```

应该看到 `admin` 用户。

## 6. 中间件启动方案

### 6.1 Docker Compose 说明

仓库提供了部分中间件的 Docker Compose：

```text
resources/docker/rocketmq-stack-5.2.0.compose.yaml    # RocketMQ
resources/docker/milvus-stack-2.6.6.compose.yaml      # Milvus + RustFS + etcd
resources/docker/lightweight/milvus-stack-2.6.6.compose.yaml  # 轻量版 Milvus
```

**为什么没有一键全量 Compose**：

仓库没有发现一份同时覆盖 PostgreSQL、Redis、RocketMQ、RustFS、Milvus 的全量一键 Compose。PostgreSQL 和 Redis 需要你自己准备（Docker 或本机安装）。

### 6.2 PostgreSQL 和 Redis（手工准备）

**PostgreSQL 推荐用 Docker**：

```bash
docker run -d \
  --name ragent-postgres \
  -e POSTGRES_USER=postgres \
  -e POSTGRES_PASSWORD=postgres \
  -e POSTGRES_DB=ragent \
  -p 5432:5432 \
  -v ragent_pg_data:/var/lib/postgresql/data \
  ankane/pgvector:latest
```

**为什么用 `ankane/pgvector`**：这个镜像已经内置 pgvector 扩展，省去手动安装步骤。

**Redis 推荐用 Docker**：

```bash
docker run -d \
  --name ragent-redis \
  -p 6379:6379 \
  -v ragent_redis_data:/data \
  redis:7-alpine
```

**验证 PostgreSQL**：

```bash
psql -h 127.0.0.1 -U postgres -d ragent -c "SELECT 1;"
```

**验证 Redis**：

```bash
redis-cli ping
```

期望返回 `PONG`。

### 6.3 RocketMQ（Docker Compose）

**为什么需要**：知识库文档上传后，分块任务通过 RocketMQ 异步触发。

**启动命令**：

```bash
cd resources/docker
docker compose -f rocketmq-stack-5.2.0.compose.yaml up -d
```

**验证**：

```bash
# 查看 NameServer 日志
docker logs rmqnamesrv

# 查看 Broker 日志
docker logs rmqbroker
```

NameServer 健康后，日志中应出现 `The Name Server boot success`。

Dashboard 访问：`http://localhost:8082`

### 6.4 RustFS（Docker Compose）

RustFS 有两个启动方式：

**方式 A：随 Milvus Compose 一起启动**

`milvus-stack-2.6.6.compose.yaml` 中已经包含 RustFS 服务。

**方式 B：单独启动 RustFS**

```bash
docker run -d \
  --name ragent-rustfs \
  -p 9000:9000 \
  -p 9001:9001 \
  -e RUSTFS_ACCESS_KEY=rustfsadmin \
  -e RUSTFS_SECRET_KEY=rustfsadmin \
  rustfs/rustfs:1.0.0-alpha.72 \
  --address :9000 \
  --console-enable \
  --access-key rustfsadmin \
  --secret-key rustfsadmin \
  /data
```

**验证**：

```bash
curl http://localhost:9000/
```

如果返回 XML 或 200，说明启动成功。控制台：`http://localhost:9001`。

### 6.5 Milvus（可选）

**为什么可选**：默认 `rag.vector.type=pg`，不学 Milvus 链路时不用启动。

**启动命令**：

```bash
cd resources/docker
docker compose -f milvus-stack-2.6.6.compose.yaml up -d
```

**验证**：

```bash
# 查看 Milvus 健康
curl http://localhost:9091/healthz
```

Attu 管理界面：`http://localhost:8000`

### 6.6 各中间件验证方式汇总

| 中间件 | 验证命令 | 成功标志 |
|---|---|---|
| PostgreSQL | `psql -h 127.0.0.1 -U postgres -d ragent -c "SELECT 1;"` | 返回 1 |
| pgvector | `SELECT * FROM pg_extension WHERE extname='vector';` | 有 vector 行 |
| Redis | `redis-cli ping` | 返回 PONG |
| RocketMQ NameServer | `docker logs rmqnamesrv` | 看到 boot success |
| RustFS | `curl http://localhost:9000/` | 返回 200/XML |
| Milvus | `curl http://localhost:9091/healthz` | 返回 healthy |

## 7. 启动 MCP 服务

### 7.1 启动类路径

```text
mcp-server/src/main/java/com/nageoffer/ai/ragent/mcp/McpServerApplication.java
```

### 7.2 Maven 命令

```bash
./mvnw -pl mcp-server spring-boot:run
```

**解释**：

- `-pl mcp-server`：只构建 mcp-server 模块。
- `spring-boot:run`：启动 Spring Boot 应用。

### 7.3 端口

MCP Server 默认端口 **9099**，配置在：

```text
mcp-server/src/main/resources/application.yaml
```

### 7.4 如何判断工具服务启动成功

**看日志**：无异常，且看到类似 `Started McpServerApplication`。

**用 curl 测试 tools/list**：

```bash
curl -X POST http://localhost:9099/mcp \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": 1,
    "method": "tools/list",
    "params": {}
  }'
```

如果返回包含 `weather_query`、`ticket_query`、`sales_query` 三个工具，说明启动成功。

**初学阶段**：如果暂时不学 MCP，可以先不启动 mcp-server。

## 9. 启动前端 frontend

### 9.1 npm install

```bash
cd frontend
npm install
```

**为什么**：下载 `package.json` 中声明的依赖，包括 React、Vite、Zustand、Axios 等。

**常见问题**：

- 网络慢：配置 npm 镜像，如 `npm config set registry https://registry.npmmirror.com`。
- Node 版本不兼容：切换到 Node 18/20 LTS。

### 9.2 npm run dev

```bash
npm run dev
```

**为什么**：启动 Vite 开发服务器，默认端口 5173。

成功后会看到：

```text
  VITE v5.x.x  ready in xxx ms

  ➜  Local:   http://localhost:5173/
```

### 9.3 代理配置

前端 `vite.config.ts` 中已配置：

```typescript
server: {
  port: 5173,
  proxy: {
    "/api": {
      target: "http://localhost:9090",
      changeOrigin: true,
      secure: false
    }
  }
}
```

**含义**：

- 前端地址是 `http://localhost:5173`。
- 前端请求 `/api/xxx` 时，Vite 会代理到 `http://localhost:9090/api/xxx`。
- 后端 context-path 是 `/api/ragent`，所以真实后端路径是 `http://localhost:9090/api/ragent/xxx`。

**注意**：SSE 请求路径 `/rag/v3/chat` 不走 `/api` 前缀，Vite 配置中没有 `/rag` 代理。实际项目中前端 `useStreamResponse` 可能通过 `VITE_API_BASE_URL` 或直接请求同源路径处理 SSE。如果 SSE 请求 404，需要检查前端实际请求的 URL 和后端接口是否匹配。

### 9.4 登录页和 Network 检查

1. 浏览器访问 `http://localhost:5173`。
2. 应该跳转到登录页。
3. 用 `admin/admin` 登录。
4. 打开 DevTools → Network：
   - 应该看到 `POST /api/auth/login`；
   - 登录成功后跳转，看到 `GET /api/user/me`。

**如果 401**：

- 检查 `init_data_pg.sql` 是否执行；
- 检查 Sa-Token 配置 `token-name: Authorization`；
- 检查前端 `api.ts` 是否正确从 localStorage 读取 token 并设置 Authorization 头。

## 11. 常见报错排查表

### 11.1 数据库连不上

**现象**：后端启动报错 `Connection refused` 或 `FATAL: password authentication failed`。

**排查**：

1. PostgreSQL 是否启动：`psql -h 127.0.0.1 -U postgres -c "SELECT 1;"`
2. 数据库 `ragent` 是否创建。
3. `application.yaml` 中 `spring.datasource.username/password/url` 是否正确。
4. pgvector 扩展是否安装：`SELECT * FROM pg_extension WHERE extname='vector';`

### 11.2 Redis 连不上

**现象**：报错 `Unable to connect to Redis`。

**排查**：

1. Redis 是否启动：`redis-cli ping`。
2. `spring.data.redis.host/port/password` 是否正确。
3. 如果 Redis 没有密码，删除或清空 `password` 字段。

### 11.3 RocketMQ 连不上

**现象**：日志中出现 `connect to <127.0.0.1:9876> failed`。

**排查**：

1. NameServer 是否启动：`docker logs rmqnamesrv`。
2. `rocketmq.name-server` 配置是否正确。
3. 如果只是学习 RAG 主流程，知识库分块失败不影响聊天；如果学习入库流程，必须启动 RocketMQ。

### 11.4 RustFS 连不上

**现象**：上传文档失败，日志中出现 S3/RustFS 连接错误。

**排查**：

1. RustFS 是否启动：`curl http://localhost:9000/`。
2. `rustfs.url/access-key-id/secret-access-key` 是否正确。
3. RustFS 桶是否创建（Ragent 启动时可能自动创建，也可能需要手动创建，需实际验证）。

### 11.5 模型 API Key 缺失

**现象**：提问后立即失败，日志中出现 `401 Unauthorized` 或 `API key not valid`。

**排查**：

1. 检查 `ai.providers.bailian.api-key` 等是否配置。
2. 检查环境变量 `BAILIAN_API_KEY` / `SILICONFLOW_API_KEY` / `AIHUBMIX_API_KEY` 是否设置。
3. 临时绕过：把 `rag.rerank.enabled` 设为 `false`，并只保留 Ollama 候选（如果本地有 Ollama）。

### 11.6 Embedding 维度不匹配

**现象**：入库或检索时报 `expected 1536 dimensions, not XXXX`。

**排查**：

1. `rag.default.dimension` 是否等于当前 Embedding 模型输出维度。
2. `t_knowledge_vector.embedding vector(1536)` 是否与配置一致。
3. 如果换过 Embedding 模型，可能需要重建向量。

### 11.7 401 / 403

**现象**：前端请求返回 401 或“未登录”。

**排查**：

1. `init_data_pg.sql` 是否执行，是否有 `admin` 用户。
2. 登录接口是否成功返回 token。
3. 后续请求是否携带 `Authorization` 头。
4. Sa-Token 的 `token-name` 是否为 `Authorization`。

### 11.8 前端代理失败

**现象**：前端页面空白，Network 中接口 404 或连不上。

**排查**：

1. 后端是否启动在 9090。
2. `vite.config.ts` 中 `proxy['/api'].target` 是否指向 `http://localhost:9090`。
3. 浏览器请求 URL 是否以 `/api` 开头。
4. 后端 `server.servlet.context-path` 是否为 `/api/ragent`。

### 11.9 SSE 无响应

**现象**：提问后页面一直 loading，EventStream 没有事件。

**排查**：

1. 模型 API Key 是否配置。
2. 后端日志是否有 `first packet timeout` 或模型调用异常。
3. SSE 请求 URL 是否正确（注意 context-path）。
4. 是否被浏览器插件或代理拦截。

### 11.10 MCP 工具列表为空

**现象**：问天气问题时没有调用工具，返回“未检索到”或通用回答。

**排查**：

1. MCP Server 是否启动在 9099。
2. `rag.mcp.servers` 配置是否正确。
3. 意图树中是否配置了天气意图节点和 `mcp_tool_id=weather_query`。
4. 用 curl 测试 `tools/list` 是否能返回三个工具。

## 12. 初学者推荐启动顺序

### 12.1 不要一口气启动全部

很多初学者一上来就把所有中间件、后端、前端、MCP 全启动，然后同时排查十几个问题，非常痛苦。

**推荐分层验证**：

### 阶段 1：数据库 + Redis + 后端

1. 启动 PostgreSQL + pgvector。
2. 启动 Redis。
3. 执行 `schema_pg.sql` + `init_data_pg.sql`。
4. 启动后端 `RagentApplication`。
5. 验证：后端日志无数据库/Redis 连接异常。

### 阶段 2：前端

1. `cd frontend && npm install && npm run dev`。
2. 访问 `http://localhost:5173`。
3. 用 `admin/admin` 登录。
4. 验证：能看到页面，Network 中 `/api/auth/login` 成功。

### 阶段 3：模型

1. 配置至少一个模型供应商的 API Key（如百炼或 SiliconFlow）。
2. 在聊天页提问“你好”。
3. 验证：能看到 SSE 流式回答，数据库 `t_message` 有记录。

### 阶段 4：文档上传

1. 启动 RocketMQ。
2. 启动 RustFS。
3. 在管理后台创建知识库并上传一个 txt 文件。
4. 点击“分块”。
5. 验证：`t_knowledge_document`、`t_knowledge_chunk`、`t_knowledge_vector` 有数据。

### 阶段 5：MCP

1. 启动 mcp-server。
2. 在管理后台配置天气意图节点。
3. 提问“北京今天天气怎么样”。
4. 验证：返回天气信息，`t_rag_trace_node` 中有 MCP 相关节点。

### 阶段 6：Milvus（可选）

1. 如果学到向量存储切换，启动 Milvus。
2. 改 `rag.vector.type=milvus`。
3. 重新上传文档验证 Milvus 链路。
