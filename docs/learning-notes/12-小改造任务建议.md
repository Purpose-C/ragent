# 小改造任务建议

> 本章只给方案、提示词模板和回滚建议，不实际修改业务源码。目标是让你在学习过程中，通过“设计小改造”来反向验证自己对源码的理解程度。

---

## 改造原则

1. **每次只做一个任务**，不要同时改多个模块。
2. **先写方案和测试，再改代码**。
3. **最小修改**：不顺手重构、不调整无关格式。
4. **本地分支实验**：不要在 `main` 上直接改。
5. **回滚优先用 `git revert`**，不要用 `git reset --hard`。
6. **绝不把真实 API Key、密码、token 写进代码或文档**。

## T1：入库节点日志增加 taskId 与耗时

**改造目标**：在 `IngestionEngine` 的节点执行日志中增加 `taskId` 和 `durationMs`，方便按任务维度检索日志。

**适合学习什么**：
- `IngestionEngine.executeNode()` 的日志输出位置。
- `IngestionContext` 如何保存 `taskId`。
- logback 占位符和 MDC 的基础用法。

**涉及文件**：
- `bootstrap/src/main/java/com/nageoffer/ai/ragent/ingestion/engine/IngestionEngine.java`

**涉及方法**：
- `executeNode(IngestionContext, NodeConfig)` 中三处 `log.info()` / `log.error()`。

**是否改数据库**：否。

**是否改前端**：否。

**风险点**：
- 日志输出格式变化可能影响 ELK 解析规则（如果团队有日志收集）。
- 增加 `taskId` 后日志行变长，大并发时日志量变大。

**测试方法**：
1. 在 Ingestion 页面创建一个任务。
2. 查看后端日志，确认每条节点日志包含 `taskId` 和 `durationMs`。
3. 分别测试成功节点和失败节点。

**回滚方式**：
- 还原 `IngestionEngine.java` 中的日志行改动。

**完成标准**：
- 成功/失败节点日志都包含 `taskId` 和耗时。
- 不引入新的异常。

**Codex 改造方案提示词模板**：

```text
请只修改 bootstrap/src/main/java/com/nageoffer/ai/ragent/ingestion/engine/IngestionEngine.java。

目标：在 IngestionEngine.executeNode() 的三处日志输出中，增加 taskId 和 durationMs。

要求：
1. 从 context 中读取 taskId；如果为空，显示为 "-".
2. 日志格式为："[taskId={}] 节点 {} 执行完成，耗时 {}ms: {}".
3. 失败日志也要包含 taskId、nodeId、durationMs。
4. 不要修改其他方法或类。
5. 不要引入新的依赖。

完成后说明修改了哪几行。
```

## T2：空检索时给出更明确提示

**改造目标**：当 `RetrievalContext.isEmpty()` 为 true 时，根据当前是知识库为空还是无匹配 chunk，返回不同的提示文案。

**适合学习什么**：
- `StreamChatPipeline.handleEmptyRetrieval()` 的分支。
- `RetrievalContext` 的结构。
- 如何区分“知识库没有文档”和“有文档但无匹配”。

**涉及文件**：
- `bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/service/pipeline/StreamChatPipeline.java`
- `bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/retrieve/RetrievalContext.java`

**涉及方法**：
- `handleEmptyRetrieval(StreamChatContext)`
- `RetrievalContext.isEmpty()`

**是否改数据库**：否。

**是否改前端**：否。

**风险点**：
- 改变前端预期文案，需要同步更新前端测试或截图。
- 提示信息不能泄露内部知识库结构（如“知识库 A 为空”）。

**测试方法**：
1. 新会话提问知识库外的问题，观察提示。
2. 空知识库中提问，观察提示。
3. 有文档但问题无关，观察提示。

**回滚方式**：
- 还原 `StreamChatPipeline.handleEmptyRetrieval()` 的提示文案。

**完成标准**：
- 三种场景返回三种不同文案。
- 文案不泄露内部信息。

**Codex 改造方案提示词模板**：

```text
请只修改 StreamChatPipeline.handleEmptyRetrieval() 方法。

目标：根据知识库是否有文档、是否有匹配 chunk，返回不同提示。

要求：
1. 如果当前知识库没有任何文档，提示用户先上传文档。
2. 如果有文档但无匹配 chunk，提示换种问法。
3. 文案不能泄露知识库 ID、chunk 数量等内部信息。
4. 不要修改其他方法或类。

完成后给出三种场景的测试用例。
```

## T4：启动时校验模型配置

**改造目标**：在 Spring Boot 启动时校验 `ai.chat.candidates`、`ai.embedding.candidates`、`ai.rerank.candidates` 的 API Key 是否配置，未配置时给出明确错误。

**适合学习什么**：
- Spring Boot `CommandLineRunner` 或 `ApplicationRunner`。
- `@ConfigurationProperties` 绑定。
- 候选模型配置结构。

**涉及文件**：
- 新增：`bootstrap/src/main/java/com/nageoffer/ai/ragent/bootstrap/ModelConfigValidator.java`
- `bootstrap/src/main/resources/application.yaml`（仅用于理解，不改）
- `infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/config/AIModelProperties.java`

**涉及方法**：
- 新增的 `run(ApplicationArguments)` 方法。

**是否改数据库**：否。

**是否改前端**：否。

**风险点**：
- 校验过严会导致本地无模型时无法启动，影响只学习后端流程的人。
- 不同 provider 对 API Key 的要求不同（如 Ollama 不需要）。

**测试方法**：
1. 删除某个候选的 API Key 环境变量，启动后端，观察错误提示。
2. 恢复 API Key 后，启动成功。

**回滚方式**：
- 删除新增的 `ModelConfigValidator.java` 和自动装配注册。

**完成标准**：
- 缺少 API Key 时启动失败并指出具体配置键。
- Ollama 等无需 Key 的 provider 不报错。

**Codex 改造方案提示词模板**：

```text
请在 bootstrap 模块新增一个启动校验类。

目标：Spring Boot 启动时校验 ai.chat / ai.embedding / ai.rerank 的候选模型配置。

要求：
1. 新增 ModelConfigValidator 实现 CommandLineRunner。
2. 只校验需要 API Key 的 provider（如 bailian、siliconflow、aihubmix），ollama 和 noop 跳过。
3. 未配置 API Key 时抛异常并停止启动，错误信息包含 provider 和 model id。
4. 不要修改现有业务代码，只新增文件和在自动配置中注册。

完成后给出测试步骤。
```

## T5：Trace 节点增加检索数量

**改造目标**：在 RAG Trace 的 `RETRIEVE` 节点 `extra_data` 中记录召回数量、去重后数量、Rerank 后数量。

**适合学习什么**：
- `RagTraceAspect` 注解节点和 `extra_data` 用法。
- `RetrievalEngine` 返回的 `RetrievalContext`。
- 如何在方法内部修改 Trace 节点信息。

**涉及文件**：
- `bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/retrieve/RetrievalEngine.java`
- `bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/aop/RagTraceAspect.java`（可能需要新增能力）

**涉及方法**：
- `RetrievalEngine.retrieve()`

**是否改数据库**：否（只改 extra_data JSON 内容）。

**是否改前端**：否（前端已展示 extra_data）。

**风险点**：
- `RetrievalContext` 可能不包含中间数量，需要计算。
- extra_data 体积增加，但对单节点影响很小。

**测试方法**：
1. 提问后查 `t_rag_trace_node`，看 `RETRIEVE` 节点 extra_data 是否包含数量。
2. 对比 `RetrievalContext` 中实际 chunk 数量。

**回滚方式**：
- 还原 `RetrievalEngine.retrieve()` 中写入 extra_data 的代码。

**完成标准**：
- `RETRIEVE` 节点 extra_data 包含召回数、去重后数、Rerank 后数。
- 不影响正常问答流程。

**Codex 改造方案提示词模板**：

```text
请修改 bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/retrieve/RetrievalEngine.java。

目标：在 retrieve() 方法结束时，把召回数量、去重后数量、Rerank 后数量写入 Trace 节点 extra_data。

要求：
1. 获取当前 RagTraceContext 中的 nodeId（如果存在）。
2. 使用 RagTraceRecordService 更新该节点的 extra_data。
3. JSON 字段名为 retrieveCount、dedupCount、rerankCount。
4. 如果 Trace 未启用，不执行任何操作。
5. 不要修改 RetrievalEngine 的核心检索逻辑。

完成后给出验证 SQL。
```

## T7：Pipeline 节点增加自定义超时

**适合学习什么**：
- `NodeConfig` 结构。
- `IngestionEngine.executeNode()` 执行流程。
- Java `Future` / `CompletableFuture.get(timeout, unit)`。
- `t_ingestion_pipeline_node` 表结构。

**涉及文件**：
- `bootstrap/src/main/java/com/nageoffer/ai/ragent/ingestion/domain/pipeline/NodeConfig.java`

- `resources/database/schema_pg.sql` 或新增 upgrade 脚本

**涉及方法**：
- `IngestionEngine.executeNode()`

**是否改数据库**：是（`t_ingestion_pipeline_node` 增加 `timeout_ms` 字段）。

**是否改前端**：是（Pipeline 编辑页面增加超时输入框）。

**风险点**：
- 超时后线程中断可能留下未完成副作用（如已写入 RustFS 的对象）。
- 节点类型不同，合理超时时间不同，需要默认值。

**测试方法**：
1. 创建一个 Pipeline，给 Parser 节点设置 1ms 超时。
2. 执行任务，确认节点失败并记录超时错误。
3. 恢复合理超时，任务成功。

**回滚方式**：
- 还原代码。
- 回滚数据库字段（保留空值兼容）。

**完成标准**：
- 节点支持配置超时。
- 超时后任务状态为 failed，错误信息明确。
